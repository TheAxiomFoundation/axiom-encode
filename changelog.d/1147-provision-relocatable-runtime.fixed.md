`scripts/provision_verification_supervisor.py` now provisions a trusted runtime whose launcher interpreter loads nothing from the user-writable source prefix. The load-bearing guarantee is empirical: the provisioned interpreter is run under a minimal environment and, from a non-empty `/proc/self/maps`, is required to map its own binary from the runtime (non-vacuity, for both shared and static-libpython builds), to map NO file under the source prefix, and to map any `libpython` only from inside the root-owned runtime — holding even under a hostile `LD_LIBRARY_PATH`. (The root-owned system loader and libc legitimately map from `/lib`/`/usr/lib`; the probe bounds the launcher's own load-time closure, not C-extensions the encoder imports later, which are covered by the run-path repin plus root-ownership.) The check observes exactly what the loader does, with no dependence on hand-parsed ELF metadata. Supporting static passes make that outcome reachable and durable: refuse system-python and oversized/symlink-escaping prefixes before copying (the first hosted-runner dispatch copytreed `/usr` because `sudo` secure_path resolved the system interpreter); purge every importable/path-configuration startup carrier after the staged site-packages swap — `.pth`/`._pth`/`sitecustomize`/`usercustomize` in all module forms, `pybuilddir.txt`, and the canonical stdlib `pythonXY[t].zip` across `PLATLIBDIR` variants; and repin every ELF whose `DT_RPATH`/`DT_RUNPATH` (via `patchelf --print-rpath`) escapes the runtime after per-object `$ORIGIN` expansion (both lexical and symlink-resolved containment) to a forced `DT_RPATH` at `<runtime>/lib` that `LD_LIBRARY_PATH` cannot outrank, re-reading to confirm. A new `provision-selftest` workflow exercises the production shape (sudo + resolved toolcache interpreter + `--require-prefix-under` + real patchelf), the self-containment probe, and the fail-closed refusal paths on the hosted Ubuntu image (3.13 + 3.14). Hostile ELF metadata in a pinned dependency is not statically audited — such a dependency already executes its own code inside the signer, so the trust boundary is dependency provenance plus the caller's root-ownership of the finished tree, which the supervisor re-verifies.
