`scripts/provision_verification_supervisor.py` now provisions a genuinely self-contained trusted runtime. The load-bearing guarantee is empirical: the provisioned launcher interpreter is run under a minimal environment and `/proc/self/maps` is required to show it maps `libpython` and all code from inside the root-owned runtime and nothing from the (user-writable) source prefix, even under a hostile `LD_LIBRARY_PATH` — a check that observes exactly what the loader does. Supporting static passes make that outcome reachable and durable: refuse system-python and oversized/symlink-escaping prefixes before copying (the first hosted-runner dispatch copytreed `/usr` because `sudo` secure_path resolved the system interpreter); purge every importable/path-configuration startup carrier after the staged site-packages swap — `.pth`/`._pth`/`sitecustomize`/`usercustomize` in all module forms, `pybuilddir.txt`, and the canonical stdlib `pythonXY[t].zip` across `PLATLIBDIR` variants; and repin every ELF whose `DT_RPATH`/`DT_RUNPATH` (via `patchelf --print-rpath`) escapes the runtime after per-object `$ORIGIN` expansion (both lexical and symlink-resolved containment) to a forced `DT_RPATH` at `<runtime>/lib` that `LD_LIBRARY_PATH` cannot outrank, re-reading to confirm. A new `provision-selftest` workflow exercises the production shape (sudo + resolved toolcache interpreter + `--require-prefix-under` + real patchelf), the self-containment probe, and the fail-closed refusal paths on the hosted Ubuntu image (3.13 + 3.14). Hostile ELF metadata in a pinned dependency is not statically audited — such a dependency already executes its own code inside the signer, so the trust boundary is dependency provenance plus the caller's root-ownership of the finished tree, which the supervisor re-verifies.
