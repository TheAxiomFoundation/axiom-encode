`scripts/provision_verification_supervisor.py` now provisions a genuinely self-contained trusted runtime, hardened over a multi-round cross-model review of the key-custody surface. It refuses system-python and oversized/symlink-escaping prefixes before copying (the first hosted-runner dispatch copytreed `/usr` because `sudo` secure_path resolved the system interpreter); purges every importable/path-configuration startup carrier — `.pth`/`._pth`/`sitecustomize`/`usercustomize` in all module forms, `pybuilddir.txt`, and the canonical `pythonXY.zip` — after the staged site-packages swap so the launcher's `-I` start executes none of them; parses each ELF's loader-facing program headers (`PT_INTERP`, `PT_DYNAMIC`) directly rather than trusting patchelf's section view, refusing any non-64-bit-LE ELF and any `DT_RPATH`/`DT_RUNPATH`/path-bearing `DT_NEEDED`/`DT_AUDIT`/`DT_DEPAUDIT`/`DT_FILTER`/`DT_AUXILIARY` that escapes the runtime after per-object `$ORIGIN` expansion (both lexical and symlink-resolved containment), relocating escaping run paths to the pinned `<runtime>/lib` as a forced `DT_RPATH` and re-parsing to confirm; and requires `PT_INTERP` to be a root-owned, non-writable system-loader path. A `/proc/self/maps` probe on the launcher interpreter empirically confirms it maps `libpython` and all code from inside the runtime and nothing from the source prefix, even under a hostile `LD_LIBRARY_PATH`. A new `provision-selftest` workflow exercises the production shape and the fail-closed paths on the hosted Ubuntu image (3.13 + 3.14).
